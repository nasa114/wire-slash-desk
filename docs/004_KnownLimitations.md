# 既知の限界（公開デプロイ前チェックリスト）

対象読者: 実装エージェント（Claude Code / Codex）および本人。

このドキュメントは、現状のコードベースで意図的に許容している既知の限界を記録する。
Dev Container（squid プロキシ経由・インターネットから隔離）での個人利用を前提に
「今は許容できるが、公開デプロイ（インターネットから到達可能な環境への配置）の
前には対応を検討すべき」項目を集める。各項目は 現状 / リスク / 対応案 / トリガー条件
の形式で記す。

> **セキュリティレビュー(2026-07-17, ペネトレーションテスト)による更新**
> 公開デプロイ(OCI 等)を見据えた敵対的レビュー(Claude + Codex + サブエージェントの
> 独立監査)を実施。以下を **TDD で対応済み**:
> - **collector 経路の SSRF(新規発見・本ドキュメント未記載だった "8件目")**: feed 取得が
>   SSRF ガードなし・リダイレクト自動追従だった問題を修正(§8 参照)。`fetch_article_content`
>   と同等の多層防御(毎ホップの公開アドレス検査・手動リダイレクト・回数制限・応答サイズ上限)を
>   collector にも適用。
> - **§7 `/login` レート制限**: 実装済み(下記 §7 を参照)。
> 静的解析で SQLi / XSS / CSRF / 認証境界 / 本文非保存の不変条件 / ユーザー列挙 /
> タイミング差は「悪用不可・堅牢」と確認済み。残る §1〜§6 は引き続き公開デプロイ前の検討事項。

## 1. DNS rebinding（TOCTOU: SSRF 検査と実接続のあいだの名前解決不一致）

- **現状**: `assertPublicHttpUrl`（`src/server/ssrf.ts`）は URL のホスト名を DNS で
  解決し、全アドレスが公開 IP であることを確認してから処理を進める。しかし実際の
  HTTP 接続（`fetchFn` 経由の `fetch`）はその後、undici が**独自に再度名前解決**を
  行う。この2回の名前解決の間に DNS レコードが書き換えられれば（DNS rebinding）、
  検査時は公開 IP、接続時はプライベート IP という状況が作れる（Time-of-check to
  time-of-use）。
- **リスク**: 悪意あるフィード/リダイレクト先が、SSRF ガードをすり抜けて内部
  ネットワーク（DB・squid管理画面等）へのアクセスに使われる可能性。
- **対応案**: 検査で確定した IP アドレスへ実際に接続を固定するカスタム
  `dispatcher`（undici の `Agent`/`connect` オプションで解決済み IP を直接使う）を
  実装し、ホスト名の再解決を発生させない。
- **トリガー条件**: アプリをインターネットから到達可能な環境（本番デプロイ）に
  置く場合。Dev Container 内（`internal` ネットワークで DB 等が隔離済み）では
  優先度低。
- **補足(2026-07-17)**: この TOCTOU は `assertPublicHttpUrl` を使う全経路
  （`fetch_article_content` と、§8 で新たにガードを適用した collector の feed 取得）に
  共通で残る。恒久対策(検査済み IP への接続固定)は両経路で共通化して実装するのが望ましい。

## 2. `/mcp` エンドポイントに HTTPS 強制なし

- **現状**: 設計書（`docs/002_Spec.md` §7）は「Phase A: 静的Bearerトークン
  …HTTPS必須」としているが、アプリ側コード（`src/server/app.ts` /
  `src/mcp/server.ts`）には HTTPS を強制するロジック・`X-Forwarded-Proto` 等の
  検証は実装されていない。TLS 終端はデプロイ層（リバースプロキシ / PaaS）の
  責務という前提で、アプリはそれを検証しない。
- **リスク**: リバースプロキシの設定漏れ等で平文 HTTP のまま `MCP_BEARER_TOKEN`
  が露出する経路が生まれても、アプリ側では検知できない。
- **対応案**: デプロイ手順書に「TLS終端必須・平文 HTTP でのアプリ直接公開禁止」を
  明記する運用でカバーしつつ、必要であれば `Forwarded` / `X-Forwarded-Proto`
  ヘッダを検証して非 HTTPS リクエストを拒否するミドルウェアを追加する。
- **トリガー条件**: インターネットに公開デプロイする時点で、デプロイ手順書に
  明記（必須）。アプリ側検証の実装は、複数デプロイ環境でリバースプロキシ設定の
  信頼性にばらつきが出る場合に検討。

## 3. SSRF 拒否帯が IANA special-purpose レジストリを完全網羅していない

- **現状**: `PRIVATE_V4_RANGES`（`src/server/ssrf.ts`）は主要な private/loopback/
  link-local/CGNAT 帯のみを拒否している。以下は現状「公開」として扱われ、
  ブロックされない:
  - `192.0.0.0/24`（IETF Protocol Assignments）
  - `198.18.0.0/15`（Benchmarking）
  - マルチキャスト（`224.0.0.0/4`）
  - `240.0.0.0/4`（Reserved）
  - `255.255.255.255/32`（Limited broadcast）
  - その他 IANA IPv4 Special-Purpose Address Registry の小規模な予約帯
- **リスク**: これらの帯域への接続がクラウド環境のメタデータ的な用途や特殊な
  ネットワーク機器に使われている場合、限定的な SSRF 経路になり得る（一般的な
  クラウドメタデータエンドポイント `169.254.169.254` は link-local として
  既にブロック済み）。
- **対応案**: IANA レジストリの全帯域をテーブルに追加する。優先度は低いが、
  公開デプロイ前に一度レビューする。
- **トリガー条件**: インターネットに公開デプロイする前。

## 4. 本文切り詰めが UTF-8 バイト基準でなく文字数（UTF-16コード単位）基準

- **現状**: `fetchArticleContent`（`src/mcp/fetch-content.ts`）の
  `MAX_TEXT_CHARS`（10万）による切り詰めは `text.length`（JS 文字列の UTF-16
  コード単位数）で判定・`slice` している。UTF-8 バイト数ではない。マルチバイト
  文字（絵文字・サロゲートペア等）を含む本文では、切り詰め位置がサロゲート
  ペアの境目に来て不正な文字列になる可能性、および実際のバイト数が想定より
  大きくなる可能性がある。
- **リスク**: 実害は限定的（表示崩れ・わずかなサイズ超過程度）だが、
  MCP クライアント側でのレンダリング崩れの原因になり得る。
- **対応案**: サロゲートペア境界を考慮した切り詰め（`Array.from(text)` で
  コードポイント単位に分割してから `slice`）、または UTF-8 バイト長基準へ
  変更する。
- **トリガー条件**: 絵文字・特殊文字を含むフィードでの実害報告があった場合、
  または公開デプロイ前の品質レビュー時。

## 5. `TRUST_EGRESS_PROXY=true` はプロキシ側 allowlist 運用が前提

- **現状**: `TRUST_EGRESS_PROXY=true` は SSRF ガードの DNS 事前解決チェックを
  スキップし、接続先制御をプロキシ（この Dev Container では squid の
  `allowed-domains.txt`）の許可リストに完全に委譲する（`src/server/ssrf.ts`
  の `assertProxySafeHttpUrl`、IP リテラル直指定のみアプリ側で拒否）。
  起動時検証（`src/config.ts` の `validateEgressProxyTrust`）により
  `HTTPS_PROXY`/`HTTP_PROXY`/`NODE_USE_ENV_PROXY=1` の設定漏れは検出するが、
  **プロキシの allowlist 自体が正しく運用されているか（プライベート IP 帯への
  プロキシ越しアクセスを許可していないか等）はアプリの管理外**であり、
  検証できない。
- **リスク**: プロキシ側の allowlist 設定ミス（例: 内部ホスト名を誤って許可、
  ワイルドカードが広すぎる）があれば、アプリ側の SSRF 防御は実質的に
  プロキシの設定品質に依存してしまう。
- **対応案**: プロキシ設定（`squid.conf` / `allowed-domains.txt` 等）の
  レビューをデプロイ手順・変更管理に組み込む。可能であればプロキシ側でも
  RFC1918 等の内部アドレスへの CONNECT を明示的に拒否する設定を入れる。
- **トリガー条件**: `TRUST_EGRESS_PROXY=true` を使う環境（この Dev Container を
  含む）でプロキシ設定を変更するたび。インターネット公開デプロイ時は、
  可能であれば直接エグレス + DNS 検査（`TRUST_EGRESS_PROXY=false`）の構成を
  優先検討する。

## 6. `/setup`（初回セットアップ）の TOCTOU と公開前露出

- **現状**: 管理ユーザーが0件のときだけ `/setup` が開放される
  （`src/server/web.ts`）。`users.count()` の判定と `create()` の間に排他は
  なく、並行 POST が同時に通れば複数ユーザーが作成され得る。また初回起動から
  セットアップ完了までの間は、到達できる誰でも管理ユーザーを作成できる。
- **リスク**: 公開網に「セットアップ未完了のまま」デプロイすると、先に到達した
  第三者が管理者になれる（self-hosted アプリ一般の first-run 問題）。
- **対応案**: セットアップ完了までは信頼ネットワーク内（ローカル/VPN）でのみ
  公開する運用を手順書に明記。厳密にやるなら `insert ... select ... where not
  exists (select 1 from users)` による原子的作成、または招待トークン方式。
- **トリガー条件**: インターネットから到達可能な環境への初回デプロイ時。

## 7. `/login` にレート制限・アカウントロックなし → **対応済み(2026-07-17)**

- **対応**: `src/server/login-throttle.ts`(`LoginThrottle`)を実装し、`POST /login`
  に配線(`src/server/web.ts`)。**ユーザー名**と**送信元 IP**の両方をキーにした
  インメモリのスライディングウィンドウ(既定: 15分・5回)で、閾値到達時は scrypt 検証を
  行わず `429 Too Many Requests` + `Retry-After` を返す(辞書攻撃の抑制に加え、
  scrypt による CPU/worker pool 枯渇 DoS も緩和)。認証成功でカウンタをリセット。
  追跡キー数に上限を設けメモリ増加を有界化。
- **回帰テスト**: `test/server/login-throttle.test.ts`(純ロジック)、
  `test/server/login-throttle-web.test.ts`(HTTP 経由)。CI(`.github/workflows/ci.yml`)で常時実行。
- **残る運用上の注意**: 送信元 IP は TCP 接続元のみを使い、`X-Forwarded-For` は既定で
  信頼しない(スプーフ回避)。信頼できるリバースプロキシ配下では、プロキシ側での
  レート制限(nginx `limit_req` / Cloudflare 等)との多層化を推奨。強いパスフレーズの
  使用は引き続き前提。単一プロセス前提のため、水平スケール時は共有ストアでの制限へ置換する。
- **受容するトレードオフ(ロックアウト DoS)**: ユーザー名キーによる制限は、攻撃者が
  正当なユーザー名(単一オーナー運用では推測容易)に対して意図的に失敗を送り続けることで、
  オーナー自身を一時的(既定 15 分)にログイン不能にできる副作用を伴う。リバースプロキシ
  配下では全接続が同一 IP キーに集約されるため、IP キーの飽和でも同様に起こり得る。これは
  総当たり防御(scrypt 枯渇 DoS の緩和を含む)と表裏の一般的トレードオフであり、単一オーナー
  前提では「一時的な妨害」に留まり資格情報の危殆化には至らないため**受容する**。恒久的に
  無効化されるわけではなく、ウィンドウ経過で自然回復する。より強い緩和が必要になった場合は
  段階的遅延 / CAPTCHA / プロキシ層での送信元別制限を検討する。

## 8. collector の feed 取得に SSRF ガードがなかった → **対応済み(2026-07-17)**

- **背景(発見された脆弱性)**: `fetch_article_content`(MCP 本文取得)は多層 SSRF 防御を
  持つ一方、**collector の feed 取得(`src/collector/collector.ts`)は SSRF 検査ゼロ・
  `redirect` 未指定(= 自動追従)** だった。悪意ある管理者を必要とせず、**管理者が購読した
  公開フィード(信頼していた第三者サーバー)が `302 Location: http://169.254.169.254/…` /
  `http://127.0.0.1:5432/` 等を返すだけ**で、collector が内部ネットワーク(クラウド
  メタデータ・DB・内部管理画面)へ到達し得た。防御が経路間で非対称だった。
- **対応**: collector の取得を `fetch_article_content` と同方針にハードニング:
  - 毎ホップで公開アドレス検査(`assertPublicHttpUrl` / プロキシ信頼モードでは
    `assertProxySafeHttpUrl`)。プライベート宛ては `SsrfError` で拒否。
  - リダイレクトを**自動追従せず手動処理**し、各ホップで再検査。回数制限(5)。
  - 応答本文をストリームで**上限バイト数(5MB)で打ち切り**(悪意/侵害フィードによる
    メモリ枯渇 DoS 対策)。egress モードは `config.trustEgressProxy` と一致させる。
- **回帰テスト**: `test/collector/collector-ssrf.test.ts`(IP リテラル/名前解決/
  リダイレクト各ケース)。CI で常時実行。
- **残る限界**: §1 の DNS rebinding TOCTOU は collector 経路にも同様に残る(検査時と
  実接続時で undici が再解決するため)。恒久対策は §1 の対応案(検査済み IP への接続固定)を
  collector・fetch-content 双方へ共通適用すること。
