# perf-memory — 「メモリリークに見える」症状の調査

「メモリが増え続ける」「重くなる」系の報告は、真のリーク以外の原因が大半。
**ヒープとRSSを区別し、強制GC付きで計測するまでリークと呼ばない。**

## 決定的観測の取り方

1. **プロセスを隔離する**: 負荷をかけるクライアント(fetch ループ等)を被験プロセスと
   同居させない。クライアント側のアロケーションが「サーバーのリーク」に見える
2. **`--expose-gc` + `global.gc()` 後の `heapUsed`** を時系列で見る。
   - heapUsed が横ばい → リークではない(RSS の増加は V8/アロケータがページを
     OS に返さないだけの正常挙動。プラトーに達するまで増え続けて見える)
   - heapUsed が単調増加 → 本物のリーク。ヒープスナップショットで retainer を辿る
3. **入力サイズを段階的に増やして時間と RSS を測る**: 2倍で4倍遅くなるなら O(n²)。
   計算量爆発は「フリーズ + メモリ急増」としてリークと誤認されやすい

## 容疑者表(リークに見えるがリークでない順)

| 容疑者 | 見え方 | 判別法 |
|---|---|---|
| RSS だけ見ている | top で増え続ける | 強制GC後の heapUsed が横ばいなら白 |
| 計測プロセスの同居 | 負荷試験中に膨張 | クライアントを別プロセスへ |
| O(n²) アルゴリズム(ループ内 join/concat/indexOf) | 大入力でフリーズ+膨張 | サイズ2倍→時間4倍か |
| 無制限キャッシュ/Map | heapUsed 単調増加 | キーの集合が有界か確認 |
| リスナー/タイマー累積 | heapUsed 単調増加 | `process.getActiveResourcesInfo()` |
| クロージャが大オブジェクト捕捉 | スナップショットで発見 | retainer グラフ |

## 事例

### 2026-07-20: アセット minify が「メモリリーク」に見えた
- 症状報告: 「CSS や JS の圧縮をしようとするとメモリリークしているように見える」
- 反証: 隔離サーバー + 強制GCで 20,000 リクエスト後も heapUsed 26.6MB 横ばい
  (最初の同居プロセス計測では RSS 209→365MB に見えた — 容疑者表の上位2つが的中)
- 真因: `minifyCss` が `(` のたびに `out.join('')` で全出力を文字列化して `url(`
  判定 → O(n²)。実測 13KB=14ms / 100KB=604ms / 500KB=16.4s + GC churn
- 発見手順: 手順2(チェーン計測)でリークを反証 → 入力サイズ段階増で計算量爆発を検出
- 修正: join を末尾3要素の O(1) 比較へ。回帰検知として大入力の時間上限テストを追加
  (壊れた実装 24s vs 修正後 33ms — 境界 3s は環境ノイズに対して十分な余裕)
