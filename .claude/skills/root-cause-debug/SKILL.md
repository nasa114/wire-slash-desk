# /root-cause-debug — 症状駆動の根本原因調査(修正より観測)

「動くはずなのに動かない」系の障害調査すべてで実行する(Web フロー・API・DB・ビルド・
CI・分散系・パフォーマンス、ドメインを問わない)。

このスキルは 2 層構造。この SKILL.md は**抽象層**(ドメイン非依存の思考の型)。
調査対象のドメインに合う**具体層**(プレイブック)があれば必ず併読する:

- `playbooks/web-flow.md` — リダイレクト・フォーム・認証/OAuth・Cookie が絡む Web フロー

該当プレイブックが無いドメインでも抽象層だけで進められる。その場合、解決後の
昇華ステップ(手順 6)で新しいプレイブックの種を残すこと。

## 鉄則(ハードゲート)

1. **修正禁止ゲート**: 以下のどちらかを満たすまで、コードを 1 行も変更しない。
   - 自分の手で症状を再現した(curl / テスト / 再実行 / 最小 repro)
   - 仮説を一意に確定させる決定的観測を得た
   「コードを読んだらバグを見つけた」は修正の許可証ではない。それは「発火しうるバグ」で
   あって「今起きている症状の原因」の証明ではない。**「確定です」「ほぼ確定」と宣言する
   前に、その仮説が偽なら観測はどう変わるかを言えること。**

2. **反証チェック**: 新しい観測(ログ・ユーザー報告・レスポンス・計測値)が来るたび、
   必ず自問する:
   - これは現行仮説の**確証**か**反証**か?
   - この観測が取られた環境に、自分の修正は**デプロイ/反映済みか**?(未反映の修正が
     「効いた証拠」は物理的にありえない)
   反証なら仮説を破棄して振り出しに戻る。確証バイアスの典型は「期待と違う観測を
   良い兆候と読み替える」こと。

3. **観測要求の凍結**: ユーザーに決定的な観測を要求したら、**回答が来るまで次の仮説に
   飛びつかない・保険の修正を積まない**。別の情報が返ってきても、元の質問を再掲して
   回収する。

## 手順

### 1. 症状の精読(コードを開く前に)

報告を一語ずつ読む。条件・限定の言い回しは裏の意味を持つ:
- 「X の状態**だと**動いた」→ X でないと動かなかった(そこが症状)
- 「たまに」「最初だけ」「再起動すると直る」→ 状態・キャッシュ・レース・リソース枯渇の示唆
- 「何も起きない」→ どこかで無言で握り潰されている(エラーが出ない ≠ エラーがない)

不足している観測を先に列挙してから調査を始める。

### 2. チェーンの全ホップを書き出す

症状に至る処理の連鎖(リクエスト列・パイプライン・データフロー・状態遷移)を 1 本の
図にする。各ホップについて「期待値」と「実測値」の表を埋める。
**バグは期待と実測が最初に食い違うホップにある。**

実測は自分で取る。curl・ローカル起動・単体テスト・print デバッグが使えるなら、
ユーザーに聞く前に自分で実行する。自分で取れない観測(ユーザーの画面・本番ログ)だけ
をユーザーに要求する。

### 3. 値は「受け手」まで追う

境界(ホップ間)で値がどう変換されるかに注目する: 絶対/相対、encode/decode の回数、
シリアライズ往復、空文字と未指定の区別、暗黙のデフォルト、型強制。

**渡す側のコードだけ読んで受け手を読まないのが最頻の見落とし。** 値を受け取る側の
サニタイザ・バリデータ・パーサ・ミドルウェアまで必ず開く。無言で値を捨てる/丸める
コード(フォールバック、`?? デフォルト`、catch 握り潰し)は最重要容疑者。

### 4. ガードレールを「容疑者リスト」として総当たり

正常系を殺すのは、多くの場合バグではなく**意図的に振る舞いを制限する善意の機構**である:
セキュリティ機構(CSP・CSRF・CORS・サニタイザ)、バリデーション、レートリミッタ、
タイムアウト、リトライ上限、キャッシュ、フィーチャーフラグ、権限チェック、
プロキシ/ロードバランサの書き換え。

チェーンの各遷移に対し「この機構はこの遷移を殺せるか?」で 1 つずつ照合する。
ドメインのプレイブックに容疑者表があればそれを使う。

**観測の死角に注意**: 「こちらは正しい出力を返しているのに、その先が起きない」と
絞り込めたら、死んでいるのは自分の観測点の**外**(ブラウザ内、クライアント側、
相手システム、ネットワーク中間装置)。そこにはこちらのログに一切痕跡が出ない
ブロック機構がある。観測点をそちら側に移す。

### 5. 整合性チェック → TDD で修正

真因候補が見つかったら、修正の前に整合性を確認する:
**真因は「なぜ今まで動いていたか」「なぜこの条件でだけ壊れるか」も説明できる。**
説明がつかない候補は真因ではないか、複数原因の一つにすぎない。

1. 真因を再現する**失敗するテスト**を先に書く(red を実行して確認)
2. 修正して green を確認
3. 全スイート回帰 + typecheck
4. なぜ壊れていたか・なぜこの形で直すのかをコードコメントに残す

実装が定型化できたらサブエージェントに委任してよい。ただし**真因の特定と検証は
委任前に本体で完了させる**こと。

### 6. 昇華(具体 → 抽象のループ)

解決したら、このスキルを育てる。これを飛ばすとスキルは古い事例集で止まる。

1. **事例の追記**: 該当ドメインのプレイブック末尾の事例集に、今回の症状・真因・
   「どの手順で発見できたか」を 3〜5 行で追記する。プレイブックが無いドメインなら
   `playbooks/<domain>.md` を新設する(容疑者表 + 事例 1 件から始めてよい)
2. **パターンの昇格**: 今回の失敗/成功パターンが**ドメインを問わず再発しそう**なら、
   この SKILL.md の鉄則・手順・アンチパターンに一般化して昇格させる。
   逆に、抽象層の記述が具体性を失って効かなくなっていたら、事例を 1 つ添えて具体化する
3. 昇格・追記の差分はユーザーに一言報告する(スキルが勝手に変わったと思わせない)

## アンチパターン(実セッションで観測された失敗)

- コードを読んで最初に見つけた「妥当そうなバグ」を症状の原因と断定する
- 未反映の修正を「観測が改善した理由」として説明する
- 決定的観測を要求した直後、回答を待たずに別仮説の修正を「保険で」積む
- 渡す側のコードだけ読み、受け手のサニタイザ/バリデータを開かない
- ファイルを全文 Read して制限機構の設定が目に入っているのに、自分が説明した
  フロー図と突き合わせない
- 自分のログに何も出ない = 問題なし、と解釈する(観測点の外のブロックはログに出ない)
- 一般論(「CSP はリダイレクトも殺しうる」)を知識として持ちながら、目の前の具体
  (このアプリの `formAction: ["'self'"]`)と突き合わせない — 抽象と具体の往復を怠る

## 出自

同一バグ(リモート MCP の OAuth 302 問題)に対し、コードから順方向に推測して修正を
積んだセッションは 33 分で未解決、症状から逆算してチェーンを追跡したセッションは
数分で 2 つの真因に到達した。この手順はその差分を打ち消すために両セッションの
ログ分析から抽出した。具体的な事例は `playbooks/web-flow.md` を参照。
